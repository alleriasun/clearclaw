# Spin-Out (Phase 1b) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Mid-work, an agent proposes splitting a related-but-separate strand into its own workspace; the human approves by creating the chat; the new workspace is seeded with the distilled brief automatically.

**Architecture:** A `spin_out` MCP tool registers a `PendingSpinOut` (brief + suggested name/cwd) in config — the same persistence pattern as `pendingPairings`. The human creates a new group and adds the bot (the approval act; bots cannot create chats). Onboarding's task prompt lists pending spin-outs; `workspace_create` gains an optional `spin_out_id` that claims one — after creation, the brief is delivered via `deliverToWorkspace()` with the originating workspace as peer origin, reusing the Phase 1 wire. Routing a strand to an *existing* peer needs no new machinery: that is `message_peer`.

**Tech Stack:** TypeScript (NodeNext, `.js` imports), in-process `clearclaw` MCP server, existing config store + peer delivery.

**Scope:** Tracer bullet. Programmatic chat creation (forum topics) and ephemeral worktree peers are Phase 1c. No teardown (manually created groups are persistent workspaces).

**Note on testing:** ClearClaw has no test runner; verification is `npm run check` (tsc) plus manual relay testing, the established workflow. The compiler is the test for type changes.

```typescript
// --- Spin-outs ---

addSpinOut(entry: PendingSpinOut): void {
  const data = this.read();
  data.pendingSpinOuts.push(entry);
  this.write(data);
}

listSpinOuts(): PendingSpinOut[] {
  return this.read().pendingSpinOuts;
}

removeSpinOut(id: string): boolean {
  const data = this.read();
  const idx = data.pendingSpinOuts.findIndex((s) => s.id === id);
  if (idx < 0) return false;
  data.pendingSpinOuts.splice(idx, 1);
  this.write(data);
  return true;
}
```

- [ ] **Step 1:** Add both tools after `message_peer` inside the same block:

```typescript
tool(
  "spin_out",
  "Propose splitting a related-but-separate strand of work into its own NEW workspace. Registers a pending brief; the user then creates a new group chat and adds the bot, and onboarding there offers to claim it. Write the brief as a distilled handoff: the goal plus the few specifics the new agent needs, not a context dump. (To hand a strand to an EXISTING workspace, use message_peer instead.)",
  {
    name: z.string().describe("Suggested workspace name (short, e.g. 'myapp-perf')"),
    brief: z.string().describe("Distilled brief delivered to the new workspace as its first message"),
    cwd: z.string().optional().describe("Suggested working directory, if known"),
  },
  async (args) => {
    const entry: PendingSpinOut = {
      id: crypto.randomUUID().slice(0, 8),
      fromWorkspace: self?.name ?? "unknown",
      name: args.name,
      brief: args.brief,
      suggestedCwd: args.cwd,
      createdAt: Date.now(),
    };
    this.config.addSpinOut(entry);
    await this.channel.sendMessage(chatId, `🌱 Spin-out "${args.name}" registered (${entry.id}). Create a new group, add me to it, and I'll offer to pick this up there.`);
    log.info("[tool] spin_out: %s registered from %s", entry.id, entry.fromWorkspace);
    return { content: [{ type: "text" as const, text: `Spin-out ${entry.id} registered. The user creates a new group chat and adds the bot; onboarding there claims the brief.` }] };
  },
),
tool(
  "spin_out_cancel",
  "Cancel a pending spin-out that has not been claimed yet.",
  { id: z.string().describe("Pending spin-out id") },
  async (args) => {
    const removed = this.config.removeSpinOut(args.id);
    return { content: [{ type: "text" as const, text: removed ? `Spin-out ${args.id} cancelled.` : `No pending spin-out "${args.id}".` }] };
  },
),
```

(`self` is already defined at the top of this block for `message_peer`.)

and after the existing `log.info("[tool] workspace_create: ...")` line (so creation is always logged):

```typescript
if (args.spin_out_id) {
  const pending = this.config.listSpinOuts().find((s) => s.id === args.spin_out_id);
  if (!pending) {
    return { content: [{ type: "text" as const, text: `Workspace "${args.name}" created, but no pending spin-out "${args.spin_out_id}" was found.` }] };
  }
  this.config.removeSpinOut(pending.id);
  this.deliverToWorkspace(args.name, { kind: "peer", workspaceName: pending.fromWorkspace }, pending.brief);
}
```

Delivery ordering: the brief is enqueued on this chat's queue while the task turn is still running (`state.busy`), and drains after the turn ends. As long as `task_complete` was called in the same turn (onboarding always does), the drain routes it as a workspace turn with peer origin — `[from <origin>]: <brief>` renders in the new chat and the new agent answers it. Document in ONBOARDING.md that claim → `task_complete` happen in one turn.
